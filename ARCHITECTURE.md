# Architecture

Human-written overview. `docs/ARCHITECTURE.md` is generated by docforge itself — this file documents the system from a contributor's point of view: how the pieces fit, why, and where the load-bearing decisions live.

## Layers

### 1. Static analysis (`scout/`, parallel scouts, no LLM)

Four agents that produce structured facts before any LLM is invoked.

| Module | Output | Job |
|---|---|---|
| `scout/manifest.py` | `Manifest` | languages, frameworks, entry points, license, deps, has_tests / has_ci / has_docker, public API list |
| `agents/test_scout.py` | `TestSummary` | test frameworks, case counts, per-symbol tested vs untested |
| `agents/api_scanner.py` | `list[APIRoute]` | HTTP routes (FastAPI/Flask/Express) + CLI commands (Click/Typer/argparse) |
| `agents/config_reader.py` | `ConfigSummary` | env vars (with secret redaction) + config file paths |

The first runs early as a prerequisite. The other three are parallel branches from `START` in the LangGraph supervisor — they write disjoint keys in `GraphState`, so the merge has zero conflict.

### 2. Grounding (`indexer/`)

`tree-sitter` produces one chunk per function/class plus a module-level chunk for imports/constants. Each chunk knows its `(file, line_start, line_end)`. `fastembed` runs `BAAI/bge-small-en-v1.5` over CPU; vectors live in a Chroma collection at `<repo>/.docforge/chroma/`.

The chunk format is the load-bearing decision here. Sliding-window chunks straddle function boundaries and produce hallucinations; function-level chunks give the LLM exactly the unit a claim usually refers to.

### 3. LLM agents (`agents/{reader, architect, diagrammer, writer}.py`)

Sequential chain after the parallel scouts. Each agent reads the relevant slice of `GraphState`, calls Groq (Llama 3.3 70B, or Qwen 2.5 Coder for code), parses, and writes back.

- **`reader`** runs a small bundle of RAG queries per top-level module ("what this module does", "public functions", "external deps", "entry points"), post-filters hits by file-path prefix to bias toward this module, and asks the LLM for a strict `ModuleSummary` JSON. Inner loop is parallelized over modules.
- **`architect`** synthesizes the module summaries into a structured `Architecture` (components, edges, external deps, runtime topology). Strict schema enforced in normalization — drops edges referencing missing components, normalizes topology to one of `cli / server / worker / library / hybrid`.
- **`diagrammer`** is **deterministic-first**. It renders Mermaid directly from `architecture.json` (always valid). An LLM beautify pass can refine layout, but only its output is kept if it also passes structural validation. On validation failure, the deterministic version wins. Net guarantee: docforge never emits broken Mermaid.
- **`writer`** drafts the four docs in parallel (`README.md`, `ARCHITECTURE.md`, `API.md`, `TUTORIAL.md`). Every prompt enforces the citation rule. Writer prompts also consume the parallel-scout outputs — TUTORIAL.md gets a real Setup section from ConfigReader, API.md gets a real route table from APIScanner.

### 4. Critic loop (`agents/{critic, editor}.py`)

The bit that makes docforge meaningfully different from any single-pass generator.

```
critic →  issues > 0  AND  cycles < MAX_CRITIC_CYCLES  ?  editor → critic  :  END
```

Three checks per cycle:

1. **Citation validity** (deterministic, free) — for each `[file.py:42]` parsed from each draft, verify the file exists and the line range is in-bounds. Out-of-bounds or missing files become `broken_citation` errors.
2. **Grounding** (LLM, parallel, bounded) — for each valid citation, ask the LLM whether the cited code supports the surrounding claim. Per-cycle budget capped at `MAX_GROUNDING_PER_CYCLE` so we never blow free-tier rate limits.
3. **Coverage** (deterministic, free) — extract bare symbol names from `manifest.public_api`, word-boundary-search them across drafts. Missing symbols become `missing_coverage` warnings.

Loop bound is hard. Empirically cycle 1 catches the hallucinations, cycle 2 mops up coverage misses, more cycles waste tokens.

### 5. Eval (`evals/`)

Separate code path used offline, not at generation time. `docforge-eval` runs the full pipeline against every entry in `eval/testset/`, then judges each output on five axes via **Claude Sonnet 4.6**:

- factuality — sampled citations checked claim ↔ code (LLM judge)
- coverage — ground-truth claims found in drafts (LLM judge)
- completeness — install/run/license/deps mentioned (deterministic regex)
- citation density — citations per 100 words (deterministic)
- readability — judge rates prose 1–5 (LLM)

The judge model is locked in code (`JUDGE_MODEL = "claude-sonnet-4-6"`) so the runner can't accidentally use a cheaper model and inflate scores. Output: `eval/scoreboard_data.json`. The eval-on-PR GitHub Action fails if any axis regresses > 2pp vs the baseline on `main`.

### 6. Web server (`server/`)

FastAPI + Jinja2 + vanilla CSS. The job runner (`server/jobs.py`) wraps the supervisor in a background thread and pushes per-agent events to an in-memory queue that the SSE endpoint streams to clients. No Next.js — server-side rendering plus a tiny `progress.js` consumer. Deploys as a single Docker image; Fly config in `fly.toml`.

## State shape (`agents/state.py`)

`GraphState` is a `TypedDict` (not Pydantic — LangGraph merges partial updates by key, scalar replacement / dict merge). Non-serializable handles (the `CodeIndex`) are passed via closure in `supervisor.build_graph()`, not via state. That keeps state checkpointable.

The new W4 keys (`test_summary`, `api_routes`, `config_summary`) are disjoint from the LLM-agent keys, so the parallel branches from `START` never conflict on merge.

## Parallelism

`agents/_parallel.py` is a tiny `ThreadPoolExecutor` wrapper. Two layers of parallelism:

1. **Graph-level** — `test_scout`, `api_scanner`, `config_reader` run concurrently from `START`. LangGraph handles the fan-out/fan-in.
2. **Within-agent** — `Reader` parallelizes per module; `Writer` parallelizes per doc; `Critic` parallelizes grounding judgments. Bounded by `DOCFORGE_MAX_PARALLEL` (default 4) so free-tier Groq rate limits stay happy.

Set `DOCFORGE_MAX_PARALLEL=1` to force serial execution (helpful for deterministic test ordering and cheap debugging).

## Non-obvious decisions

- **Defensive normalization on LLM outputs.** Reader/Architect/Diagrammer all normalize the LLM's JSON into the strict TypedDict shape themselves. We never trust the LLM to follow the schema — drop bad fields, supply defaults, clamp enums. The fallback path is always "honest empty" not "made-up content."
- **Critic falls back to "supported" on infra errors.** A failed LLM call during grounding shouldn't penalize the doc. Real ungrounded claims are detected; infra hiccups are silent.
- **Secret redaction happens at the scout, not at the writer.** Once a secret value is in `GraphState`, it could leak via logs, traces, or persisted JSON. `ConfigReader` redacts at the source.
- **In-memory job registry, not Redis.** Demo-grade. A Fly machine restart loses queued jobs. Persistence is a one-page swap when needed (see `JobRegistry` interface).
- **Mermaid validation without `mmdc`.** A regex/structure check (declaration present, balanced subgraphs, ≥1 node or edge) covers ~95% of real Mermaid breakage without a Node.js dependency in the Docker image.
