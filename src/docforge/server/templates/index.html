The problem with one-shot doc generators

Feed a model some files, ask for a README, hope for the best. It reads plausibly and breaks on contact with a real codebase:

Hallucinated APIs

It invents a function that sounds right. You can't tell without grepping the source yourself.

Missed subsystems

An entire module never gets mentioned because its file wasn't in the top-k most-similar chunks.

Diagrams that lie

The architecture reflects what an app like this usually looks like — not what your code actually does.

docforge's bet: a small team of focused agents, every claim grounded, and a step that actually verifies before publishing.

How it works

A LangGraph StateGraph. Three static-analysis scouts fan out in parallel from the start; the LLM agents then run as a chain, ending in a bounded critic → editor loop.

Meet the team

Nine agents, each with one job. Boring and small beats clever and broad.

3 static scouts · no LLM 5 LLM agents 1 critic loop

test_scout

Detects the test framework (pytest / unittest / jest / vitest / go / cargo), counts cases, and splits the public API into tested vs untested symbols.

agents/test_scout.py

api_scanner

Regex-scans for HTTP routes (FastAPI / Flask / Express) and CLI commands (Click / Typer / argparse). Every route carries a file:line.

agents/api_scanner.py

config_reader

Finds env-var reads and .env files. Redacts secret-looking values (API_KEY / SECRET / TOKEN…) before they touch state.

agents/config_reader.py:52

reader

Runs RAG retrieval per top-level module and writes a citation-grounded ModuleSummary. Fans out across modules concurrently.

agents/reader.py

architect

Rolls summaries into a structured architecture — components, edges, external deps, runtime topology. Drops edges that point at non-existent components.

agents/architect.py

diagrammer

Renders Mermaid deterministically first (always valid). An optional LLM beautify pass is kept only if it also passes validation.

agents/diagrammer.py

writer

Drafts README, ARCHITECTURE, API, TUTORIAL concurrently. Every prompt enforces the inline [file:line] citation rule.

agents/writer.py:19

critic

Three checks: deterministic citation validity, LLM grounding (does the cited code support the claim?), and deterministic coverage of the public API.

agents/critic.py

editor

Revises only the docs with open issues. Loops back to the critic until issues hit zero or the 2-cycle cap.

agents/editor.py

There's also a single-agent baseline generator — deliberately simple — that every multi-agent improvement is measured against. agents/baseline.py

Grounded & verified — the part that matters

Every claim cites [file:line]

The Writer's prompts share one hard rule: any non-trivial statement about behaviour must end with a citation like [src/api.py:42] or a range, using citations that actually appear in the retrieved context.

Chunks are stored keyed by their exact line range, so a citation is a promise you can click through and check.

The critic loop

  • 1 Citation validity — does the file exist? Is the line range in bounds? (deterministic, free)
  • 2 Grounding — does the cited code actually support the claim? (LLM, parallel, capped at 20/cycle)
  • 3 Coverage — is every public symbol mentioned somewhere? (deterministic, word-boundary)

Flagged issues go to the Editor, which revises and re-submits — bounded to 2 cycles so it always terminates. agents/critic.py

Held to a real eval

Most doc generators show one cherry-picked output. docforge scores itself on five axes, with the judge model locked in code to claude-sonnet-4-6 so nobody can swap in a cheaper model and inflate the numbers. evals/judge.py:27

    FAQ

    Does it work on big repos?

    Scout enforces a file budget, the Reader gets per-module RAG slices, and the Architect sees summaries only. ~8k LOC indexes in minutes; bump RAM for >50k LOC. Truncation is reported honestly in the output.

    What languages?

    Tree-sitter handles Python, TypeScript, and JavaScript at function granularity. Everything else falls back to a sliding-window chunker, so any repo produces output — with a partial-support note.

    Will it leak my secrets?

    No. ConfigReader redacts values for secret-looking names, and every value from a live .env is redacted regardless — only the variable's presence is recorded. agents/config_reader.py:52

    How is this different from asking an LLM for a README?

    The grounding ([file:line] on every claim), the critic loop that verifies those citations against the actual code, and a measurable eval with a locked judge. A single LLM call gives you none of those.
